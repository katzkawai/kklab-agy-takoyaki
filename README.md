# 蛸ノ灯 (Takonohi) - 極上たこ焼き公式サイト

「外はカリッ、中はトロッ。」
極上の真蛸と秘伝の出汁生地、そして熟練の職人技が織りなす究極のたこ焼きを提供する「蛸ノ灯（たこのひ）」の公式サイト兼、インタラクティブ注文シミュレーターです。

---

## 📸 ビジュアルデザイン

夜の賑やかな屋台を連想させる「ネオ和風」デザイン。
黒とダークウッドを基調とし、赤提灯の暖かな光やゴールドのネオンライト、そして湯気のゆらめきによる上質でシズル感溢れる体験を提供します。

---

## ✨ 主な機能

1. **シズル感のある湯気エフェクト**
   - ヒーロー画像の焼きたてたこ焼きと、シミュレーターの木皿からCSS粒子アニメーションで温かい湯気が立ち上ります。
2. **お品書き（Menu）**
   - 定番、ねぎ塩マヨ、明太あぶりチーズ、極み素焼きの4種類。各カードはホバー時にたこ焼き球体が滑らかにズーム・回転するアニメーション付きです。
3. **トッピングシミュレーター**
   - ベースのたこ焼きを選択すると、そのベースに合ったトッピングが自動でセットされます。
   - トッピングチェックボックス（ソース、マヨネーズ、青のり、かつお節、九条ねぎ、炙りチーズ、明太子）をON/OFFすると、**3D風の皿の上のたこ焼き（6個）にリアルタイムでトッピング具材が重ねられ、金額も自動で再計算**されます。
4. **スライドイン注文カート**
   - 自分好みにカスタマイズしたたこ焼きを「注文カート」に入れると、画面右側からカートがスライドイン。
   - LocalStorageと連動しているため、ブラウザをリロードしても選択したカート情報は失われません。
5. **模擬注文と紙吹雪演出**
   - カートから「模擬注文を確定する」を実行すると、キャンバス制御によるカラフルな紙吹雪（Confetti）が画面全体に舞い散り、感謝のモーダル画面が表示されます。

## 📦 `npm run build` の役割（ビルドの仕組み）

本プロジェクトで `npm run build` を実行すると、裏ではビルドツール（**Vite**）が動作し、開発用に書いたソースコードを**「本番のウェブサーバーで最も速く、安全に動く状態」に最適化**します。

具体的には、以下の処理を一瞬で行っています：

1. **ファイルの軽量化 (Minify):** コード内の不要な改行、空白、コメントをすべて削除し、JavaScriptの変数名を短い英字に自動変換することで、ファイルサイズを極限まで小さくします。
2. **コードの結合 (Bundling):** 開発時に整理・分割していたCSSやJavaScriptファイルを数本にまとめ、ブラウザが一度に効率よくロードできるようにします。
3. **キャッシュ対策 (Hashing):** 出力されるCSSやJSファイル名にランダムな文字列（例: `index-a1b2c3d4.js`）を付与します。コード変更時のみファイル名が変わるため、ブラウザの古いキャッシュによる誤表示を防ぎます。
4. **アセット最適化 & コピー:** 画像ファイルなどを公開場所へ適切にコピーし、本番用のアクセスパスへ自動的に書き換えます。

ビルドが完了すると、プロジェクト内に **`dist` (配布用) フォルダ** が作成されます。本番サーバーには、この **`dist` フォルダの中身のみ** をアップロードしてサイトを公開します。

## 🌐 GitHub Pages への公開方法 (デプロイ)

本プロジェクトは **GitHub Actions** を利用して、GitHub Pages 上に自動的に公開（デプロイ）されるように構成されています。初心者の方でも以下の流れで簡単に管理できます。

### 📁 公開に必要な設定ファイル（準備済み）
以下の2つのファイルが自動デプロイの役割を担っています。
1. **[vite.config.js](file:///home/katzkawai/kklab-agy-examples/vite.config.js)**
   - GitHub PagesのURL（`https://<ユーザー名>.github.io/<リポジトリ名>/`）のサブディレクトリで、画像やスタイルが正しく読み込まれるようにベースパスを設定しています。
2. **[.github/workflows/deploy.yml](file:///home/katzkawai/kklab-agy-examples/.github/workflows/deploy.yml)**
   - コードをGitHubに送信（プッシュ）したときに、自動的にプログラムを組み立て（ビルド）、公開ページに反映させるための自動化設計図です。

### ⚙️ GitHub上でのワンタイム設定 (初回のみ)
GitHub Actionsによる自動デプロイを有効にするために、以下の設定を行ってください。

1. ご自身のGitHubリポジトリ（ `https://github.com/katzkawai/kklab-agy-examples` ）を開きます。
2. 上部メニューの ⚙️ **Settings** (設定) をクリックします。
3. 左サイドバーにある **Pages** をクリックします。
4. **Build and deployment** エリアにある **Source** のプルダウンメニューを開き、デフォルトの「Deploy from a branch」から **「GitHub Actions」** に変更します。

これだけで設定は完了です！

### 🔄 日々の更新方法
今後、HTMLやCSS、JSなどのファイルを修正して公開したいときは、以下の手順のみで自動更新されます。

1. ローカルでファイルを編集します。
2. 変更をGitでコミットし、GitHubにプッシュします。
   ```bash
   git add .
   git commit -m "修正内容のメッセージ"
   git push origin main
   ```
3. プッシュすると、GitHub側で自動的にビルドと公開作業が始まります。進行状況はリポジトリの **Actions** タブから確認できます。

## 🤖 GitHub Actions の仕組みと便利な機能

### 💡 GitHub Actions とは？
GitHub Actionsは、GitHub上にコードを保存するだけでなく、 **「テスト」「ビルド」「デプロイ」などの定型的な開発作業を自動で行ってくれる強力なアシスタント機能（CI/CDツール）** です。

リポジトリ内の特定のフォルダ（`.github/workflows/`）の中に YAML 形式（`.yml`）の設計図ファイルを配置することで、GitHubが裏側で一時的な仮想サーバーを立ち上げ、指定した手順通りに作業を代行します。

---

### 🌟 便利な機能の一例：定期実行（スケジュール設定/cron）
GitHub Actionsでは、コードの変更時（プッシュ時）だけでなく、**「決まった時間や曜日に自動でタスクを実行させる（cronのような仕組み）」**ことも簡単に実現できます。

#### 設定方法
ワークフローファイルの `on:`（実行トリガー）セクションに `schedule` イベントを指定します。

```yaml
on:
  schedule:
    # 毎日 UTC 午前0時（日本時間 午前9時）に自動実行
    - cron: '0 0 * * *'
```

#### ⏱️ cron書式の一覧
```text
┌───────────── 分 (0 - 59)
│ ┌─────────── 時 (0 - 23)
│ │ ┌───────── 日 (1 - 31)
│ │ │ ┌─────── 月 (1 - 12)
│ │ │ │ ┌───── 曜日 (0 - 6) ※0は日曜日
│ │ │ │ │
* * * * *
```

#### 💡 注意点
* **UTC基準:** タイムゾーンは世界標準時（UTC）固定です。日本時間（JST）で動かしたい場合は「日本時間からマイナス9時間」で設定します（例: 朝9時 ➜ `0 0 * * *`）。
* **実行時間の遅延:** サーバーの混雑状況によって、指定時刻から数分〜数十分遅れて開始されることがあります。
